# Agent Model Override Onboarding

## ADDED Requirements

### Requirement: Model Selection step in `/onboard`

The `/onboard` skill SHALL include a "Model Selection" step positioned
between the existing Specialist Discovery step and the Summary step.
The step MUST begin with a single yes/no preamble question that gates
the per-agent prompts, so users who answer "no" proceed directly to the
Summary without seeing further model-related questions.

The step MUST NOT fire the per-agent prompts if the user answers "no"
to the preamble. When skipped, the step MUST NOT write any files and
MUST NOT modify any existing files.

#### Scenario: Step is present in the correct position

- **WHEN** a reader inspects `skills/onboard/SKILL.md`
- **THEN** the file defines a Model Selection step whose position in the
  documented Process is after Specialist Discovery and before the
  Summary and Next Steps step

#### Scenario: Preamble is a single yes/no

- **WHEN** the Model Selection step begins
- **THEN** the user is asked a single yes/no question that decides
  whether to continue into the per-agent prompts or to skip the step
  entirely

#### Scenario: Skip produces no side effects

- **WHEN** the user answers "no" to the preamble
- **THEN** no files are written under `.claude/agents/` and the existing
  plugin defaults remain in effect; the Summary explicitly states that
  no model overrides were created

### Requirement: Per-agent trade-off prompts

When the user opts into the Model Selection step, the skill SHALL ask a
bounded multiple-choice question for each of Tech Lead, Chief Architect,
and Product Owner. Each question MUST frame the choice in trade-off
terms, not in model names. The bounded answer set MUST include at least
three options: "faster and lower cost", "default", and "higher-quality
planning", in that order.

The skill MUST ask one agent's question at a time, waiting for the
answer before asking the next, consistent with the interview discipline
documented in `CLAUDE.md`.

The skill MUST NOT expose literal Claude model names to the user in the
question text. The internal mapping from trade-off answer to concrete
model name MUST be kept in a single table in `skills/onboard/SKILL.md`
so updates to the Claude model catalog are a single-place edit.

#### Scenario: Three agents are prompted

- **WHEN** the user opts into the Model Selection step
- **THEN** the skill asks three separate questions, one for Tech Lead,
  one for Chief Architect, and one for Product Owner, in that order

#### Scenario: Trade-off framing, not model names

- **WHEN** the skill asks any per-agent question
- **THEN** the prompt text names the agent and frames the choice in
  terms of speed, cost, and planning quality, and does not include the
  string "opus", "sonnet", "haiku", or any other literal Claude model
  name

#### Scenario: Answer set is bounded

- **WHEN** the skill asks any per-agent question
- **THEN** the answer set is a bounded multiple-choice set of at least
  three options that includes "faster and lower cost", "default", and
  "higher-quality planning"

#### Scenario: One question at a time

- **WHEN** the user answers one per-agent question
- **THEN** the skill asks the next per-agent question only after the
  prior answer is received; the skill does not batch the three
  questions into a single prompt

### Requirement: Override files in `.claude/agents/`

When the user selects a non-default option for an agent, the skill
SHALL write a project-local override file to
`.claude/agents/<agent-name>.md`. The override file MUST be a copy of
the current plugin agent body with the `model:` field changed to the
model that corresponds to the user's chosen trade-off. All other
frontmatter fields and the full body below the frontmatter MUST be
copied verbatim from the plugin agent file.

The override file MUST include a header comment that records:

- That the file was generated by `/onboard`.
- The path to the plugin agent file the override was copied from.
- A hash of the plugin agent body at generation time so a later run
  can detect whether the plugin body has drifted.

The skill MUST NOT modify any file under `agents/` in the plugin
directory.

#### Scenario: Override file is written for non-default answers

- **WHEN** the user selects "faster and lower cost" or "higher-quality
  planning" for Tech Lead
- **THEN** the skill writes `.claude/agents/tech-lead.md` containing
  the plugin agent body with only the `model:` field changed to match
  the chosen trade-off

#### Scenario: Default answer produces no file

- **WHEN** the user selects "default" for any agent
- **THEN** the skill does not write an override file for that agent

#### Scenario: Plugin files are not modified

- **WHEN** any agent receives an override
- **THEN** the corresponding file in the plugin's `agents/` directory
  is byte-identical before and after the skill run

#### Scenario: Header comment records provenance

- **WHEN** an override file is written
- **THEN** the first lines of the file form a header comment that
  identifies the file as generated by `/onboard`, names the source
  plugin agent file, and records a hash of the plugin agent body at
  generation time

### Requirement: Pre-existing override handling

When the skill is about to write an override to a path where a file
already exists, it SHALL detect the existing file and ask the user
whether to leave the existing file unchanged or replace it with a
freshly generated override. The skill MUST NOT overwrite a pre-existing
file without explicit user confirmation.

#### Scenario: Existing override is detected

- **WHEN** `.claude/agents/<agent-name>.md` already exists at the moment
  the skill is about to write an override for that agent
- **THEN** the skill shows the existing file's presence to the user and
  asks whether to leave it unchanged or replace it

#### Scenario: User declines replacement

- **WHEN** the user declines to replace an existing override
- **THEN** the skill does not modify the existing file and records the
  decision in the Summary

#### Scenario: User accepts replacement

- **WHEN** the user accepts replacement of an existing override
- **THEN** the skill writes the new override file, overwriting the
  prior contents, and records the action in the Summary

### Requirement: Body-hash regeneration detection

On any subsequent `/onboard` run after override files have been
generated, the skill SHALL compare the recorded body hash in each
override's header comment against a fresh hash of the current plugin
agent body. If the hashes differ, the skill MUST offer the user a
choice to regenerate the override (losing any local edits to the
override file) or to keep the current override (accepting body drift).

The skill MUST NOT regenerate an override silently. Regeneration
requires explicit user consent.

#### Scenario: Unchanged plugin body produces no prompt

- **WHEN** a generated override file's recorded body hash matches a
  fresh hash of the current plugin agent body
- **THEN** the skill does not prompt the user for regeneration for
  that agent

#### Scenario: Changed plugin body prompts the user

- **WHEN** a generated override file's recorded body hash does not
  match a fresh hash of the current plugin agent body
- **THEN** the skill shows the user the drift situation and offers a
  choice to regenerate the override or keep the current override

#### Scenario: No silent regeneration

- **WHEN** the skill detects a body-hash mismatch
- **THEN** the skill does not overwrite the override until the user
  explicitly chooses regeneration

### Requirement: README Model Selection Guidance section

The plugin's top-level `README.md` SHALL include a new subsection
titled "Model Selection Guidance" under the existing "Setting Up for
Your Project" section. The subsection MUST include one short paragraph
per agent for Tech Lead, Chief Architect, and Product Owner, describing
the trade-off using the same three-option vocabulary as the onboarding
prompt (faster and lower cost, default, higher-quality planning).

The subsection MUST state that other agents can be overridden manually
by creating a matching file in `.claude/agents/` with only the `model:`
field changed, and MUST document the manual frontmatter-edit fallback
path for environments where `.claude/agents/` overrides do not load
cleanly.

#### Scenario: Section is present in the right place

- **WHEN** a reader opens `README.md`
- **THEN** the file contains a "Model Selection Guidance" subsection
  positioned under "Setting Up for Your Project"

#### Scenario: Three agents are covered

- **WHEN** a reader reads the Model Selection Guidance subsection
- **THEN** the subsection contains at least one paragraph each for
  Tech Lead, Chief Architect, and Product Owner, each paragraph
  describing the trade-off in terms of cost, speed, and planning
  quality

#### Scenario: Trade-off vocabulary matches the prompt

- **WHEN** a reader cross-references the README subsection against the
  onboarding prompt text
- **THEN** the three trade-off labels ("faster and lower cost",
  "default", "higher-quality planning") appear in both places with the
  same spelling

#### Scenario: Manual fallback is documented

- **WHEN** a reader reads the Model Selection Guidance subsection
- **THEN** the subsection includes a path for users to override an
  agent by editing `model:` in the plugin agent file directly, with a
  warning that the edit is overwritten by plugin updates

### Requirement: Default behavior on skip is unchanged

When the user skips the Model Selection step (either by answering "no"
to the preamble or by not running `/onboard` at all), the plugin SHALL
continue to operate using the shipped default models declared in each
agent's frontmatter. No override files SHALL be created implicitly.

#### Scenario: Fresh project with skipped step

- **WHEN** a user runs `/onboard` on a fresh project and answers "no"
  to the Model Selection preamble
- **THEN** `.claude/agents/` contains no override files for the three
  prompted agents, and subsequent invocations use the shipped default
  models

#### Scenario: Project that never ran `/onboard`

- **WHEN** a project installs the plugin but never runs `/onboard`
- **THEN** the plugin's shipped default models govern every agent
  invocation and no override file is expected to exist

### Requirement: Override mechanism verification precedes writes

The skill SHALL verify that Claude Code loads a project-local
`.claude/agents/<name>.md` override before the first time it writes
override files in a project. If the verification fails, the skill MUST
fall back to emitting the exact frontmatter edit the user can apply
manually to the plugin agent file and MUST warn the user that such
edits are lost on plugin update. No override files SHALL be written in
the failed-verification path.

#### Scenario: Verification passes, overrides are written

- **WHEN** the skill's override-mechanism verification passes and the
  user has selected a non-default trade-off for at least one agent
- **THEN** the skill writes override files under `.claude/agents/`

#### Scenario: Verification fails, fallback copy is emitted

- **WHEN** the skill's override-mechanism verification fails
- **THEN** the skill does not write override files; instead it prints
  the exact `model:` frontmatter edit to apply to each affected plugin
  agent file, with a warning that the edit is overwritten by plugin
  updates

### Requirement: Summary reports overrides explicitly

The `/onboard` Summary step SHALL list which agents received override
files, which did not, and whether the Model Selection step was skipped.
The Summary MUST allow a reader to distinguish "skipped" from "answered
default for all agents" from "answered with overrides for a subset".

#### Scenario: Skipped step

- **WHEN** the user skipped the Model Selection step
- **THEN** the Summary contains a line stating that the Model Selection
  step was skipped and no override files were created

#### Scenario: Overrides created for a subset

- **WHEN** the user opted in and selected non-default trade-offs for
  some agents and "default" for others
- **THEN** the Summary lists the agents that received overrides and
  the path of each override file, and lists the agents that did not
  receive overrides

#### Scenario: All defaults

- **WHEN** the user opted in but selected "default" for every agent
- **THEN** the Summary states that every agent received the default
  trade-off and no override files were created
