# Design: Agent Model Override Onboarding

## Context

The engineering-leaders plugin ships eight agents, each with a declared
default model in its frontmatter. The current defaults are:

| Agent | Default Model |
|-------|---------------|
| chief-architect | opus |
| product-owner | opus |
| ux-strategist | opus |
| tech-lead | sonnet |
| qa-lead | sonnet |
| devops-lead | sonnet |
| agile-coach | sonnet |
| engineering-manager | sonnet |

The Tech Lead is the most load-bearing Sonnet-default agent because it owns
the Implementation Planning response mode, Phase 1 routing, and Phase 2
synthesis. The Chief Architect and Product Owner already default to Opus
but users running non-Opus main loops may want to downshift them for cost
or speed reasons. The remaining agents are less critical to plan quality
and their defaults are rarely the right thing to override first.

Claude Code's subagent loader reads agent frontmatter from plugin files
and honors the `model:` field when invoking the subagent. Claude Code also
loads agent files from the project's `.claude/agents/` directory and, when
an agent of the same name exists there, the project-local file wins. This
is the mechanism that lets users override the plugin's default model
without modifying plugin files in place. The override path is verified as
the first task in this change; if verification fails the onboarding step
falls back to emitting a manual edit instruction rather than writing
override files that would not take effect.

The `/onboard` skill is the plugin's canonical onboarding entry point. It
already follows the interview discipline documented in `CLAUDE.md`: one
question at a time, multiple-choice where bounded, every question
skippable, and a summary at the end. Adding a new step there is the most
natural place for the model-override prompt because the user is already
in the mindset of configuring the plugin for their project.

## Goals / Non-Goals

**Goals:**

- Surface the per-agent model-bias trade-off to every user during
  onboarding, without requiring them to know that agent frontmatter
  exists.
- Produce per-project override files that do not require hand-editing
  plugin files and are not clobbered by plugin updates.
- Document the trade-off in the README so the guidance survives outside
  the onboarding moment.
- Preserve existing behavior for users who skip the prompt so no current
  deployment changes unexpectedly.

**Non-Goals:**

- Changing the plugin's shipped default models. The defaults table above
  remains the source of truth.
- Forcing a choice. The prompt is skippable and the default behavior on
  skip is no override files.
- Runtime or per-story model selection. Tiered orchestration is a
  separate concern owned by `add-tech-lead-orchestration-tiers` and any
  follow-up to `/plan-implementation`.
- Cost estimation or budget tracking. The trade-off is framed
  qualitatively in the prompt and README.
- Overriding models for ad-hoc invocations or for Skills (which do not
  declare models in the same frontmatter shape as Agents).
- Covering every agent in the initial prompt. This change covers the
  three highest-leverage agents; expansion is a follow-up.

## Decisions

### D1. Per-project override files in `.claude/agents/`

The onboarding skill writes user overrides to
`.claude/agents/<agent-name>.md` in the user's project. Each override
file is a verbatim copy of the current plugin agent body with one field
changed: the `model:` value in the frontmatter. The body below the
frontmatter is copied unchanged so the agent's behavior contract is
preserved. A header comment at the top of the file records that it was
generated by `/onboard` and names the plugin agent and version it was
copied from.

**Rationale:** This is the Claude Code-sanctioned mechanism for
project-local agent customization. Users do not need to know about
`model:` or frontmatter; the skill does the copy. Plugin files are
untouched so plugin updates do not clobber user choices. The header
comment gives the user and a future maintainer a breadcrumb back to the
source.

**Alternatives considered:** Modify plugin agent files in place. Rejected
because plugin updates would overwrite the user's choice and because the
plugin directory is typically read-only or managed by the marketplace.
Also considered: write a `settings.json` override mechanism. Rejected
because settings.json is not the documented path for per-agent model
configuration in Claude Code and introduces a second source of truth.

### D2. Trade-off framing, not model names

Onboarding prompts frame each choice in user-observable terms:

> "For the Tech Lead agent, pick the bias you want for implementation
> planning: (a) faster and lower cost, (b) default, (c) higher-quality
> planning. Tech Lead is the agent that deconstructs stories, routes to
> specialists, and synthesizes their input."

The skill maps answers to concrete models internally using a small
mapping table documented in the README. The user never has to type a
model name.

**Rationale:** Model names and tiers change over time; trade-offs do
not. Framing in trade-off terms keeps the prompt comprehensible for
users new to Claude and keeps the prompt itself durable across model
generations. The internal mapping table is a single-place update when
Anthropic ships a new model.

**Alternatives considered:** Ask for a literal model name. Rejected:
users must know the current model catalog and the question becomes
stale as models evolve. Also considered: a free-text "which model"
prompt. Rejected: unbounded inputs are a source of typos and subtle
misconfiguration.

### D3. Initial coverage is Tech Lead, Chief Architect, Product Owner

The onboarding step prompts for these three agents only. Rationale,
per agent:

- **Tech Lead:** highest-leverage Sonnet default. Directly affects plan
  quality across the plugin.
- **Chief Architect:** Opus default, but users on non-Opus budgets may
  want to downshift. The architect's output directly shapes schema and
  API decisions so the trade-off is high-signal.
- **Product Owner:** Opus default with frequent invocation. The
  trade-off between faster decomposition and higher-quality story
  authorship is the most common case users ask about.

Other agents keep their current defaults and are not prompted for in
this change. A follow-up can add more agents to the prompt if usage
shows the trade-off matters for them.

**Rationale:** Keeps the onboarding question count low. Interview
discipline (`CLAUDE.md`) is "one question at a time" and bulk-prompting
eight agents degrades the experience. Three is the minimum viable set
that covers the biggest-impact trade-offs.

**Alternatives considered:** Prompt for all eight agents. Rejected: too
many questions, and most agents do not have a well-understood trade-off
to present. Also considered: prompt for Tech Lead only. Rejected: the
Chief Architect and Product Owner trade-offs are the next two most
commonly asked about and are straightforward to include without bloat.

### D4. README "Model Selection Guidance" section

A new section under "Setting Up for Your Project" in `README.md`
documents the trade-off per agent in one short paragraph. The section
explicitly names the three agents the onboarding step covers and
mentions that other agents can be overridden manually by creating
matching files in `.claude/agents/`.

**Rationale:** The onboarding prompt is ephemeral. The README is the
durable surface where users return when the prompt has passed. The
section's structure mirrors the prompt structure so a user who saw the
prompt recognizes the guidance in the README and vice versa.

**Alternatives considered:** Put the guidance inside individual agent
files. Rejected: the guidance is cross-agent and lives more naturally
in the README. Also considered: a separate `docs/model-selection.md`.
Rejected as over-structure for a short section; the README is already
the documentation home.

### D5. Skip preserves current behavior

If the user skips the onboarding step, the skill writes no override
files and the plugin's shipped defaults remain in effect. The skill's
Summary explicitly states that no overrides were created so the user
knows the plugin is running its defaults.

**Rationale:** Backward compatibility for current deployments is a
stated success metric. Silent non-action on skip is the safest default.

**Alternatives considered:** Default to "higher quality" on skip.
Rejected: silently shifts user spend without consent and surprises
users who expected to defer the decision.

### D6. Manual fallback for environments where overrides do not load

The first task in the change validates that Claude Code loads
`.claude/agents/<name>.md` overrides in the current environment. If
validation fails (for example, the override is not picked up), the
onboarding step falls back to emitting the exact frontmatter edit the
user can apply manually to `agents/<agent-name>.md` along with a
warning that the edit will be overwritten by plugin updates. The
fallback is documented in the README section.

**Rationale:** Claude Code's override behavior is currently validated
informally. If a future version of Claude Code changes that behavior,
the skill degrades gracefully rather than silently producing files
that do nothing. The fallback also gives users on older Claude Code
versions a working path.

**Alternatives considered:** Skip the manual fallback and assume
overrides always load. Rejected: silent breakage is exactly the
failure mode the project CLAUDE.md and global instructions want us to
avoid.

### D7. Regeneration on plugin update is explicit, not automatic

The skill does not watch for plugin updates and does not automatically
regenerate override files when the plugin's agent body changes. Instead,
override files include a header comment recording the plugin agent's
body hash at generation time. A future `/onboard` re-run detects a
mismatch and offers the user a choice: regenerate the override (losing
any local edits to the override) or keep the current override (accepting
body drift).

**Rationale:** Automatic regeneration would silently overwrite user
edits the first time the plugin updates. Detecting the mismatch and
asking the user preserves agency. Body-hash detection is a simple,
local mechanism that does not require network calls or plugin
introspection.

**Alternatives considered:** Regenerate silently on every `/onboard`
run. Rejected: overwrites user customization without consent. Also
considered: never regenerate. Rejected: override files drift
indefinitely and stop reflecting plugin improvements.

## Risks / Trade-offs

- **Override mechanism compatibility.** Claude Code's treatment of
  `.claude/agents/` overrides could change. **Mitigation:** The first
  task validates the mechanism in the current environment; the skill
  falls back to the documented manual edit path if validation fails;
  the README lists both paths so users on any supported Claude Code
  version have a way to proceed.
- **Default model drift.** As Anthropic ships new models the trade-off
  framing paragraph in the README can drift from the current catalog.
  **Mitigation:** The prompt is written in trade-off terms, not model
  names. The internal trade-off-to-model mapping is a single place to
  update when a new model ships. The README section points users at the
  current mapping rather than hardcoding model names in paragraphs.
- **Partial adoption.** Users who answer for one agent but skip others
  end up with a mixed state. **Mitigation:** The Summary lists which
  agents received overrides and which did not so the state is legible.
  Re-running `/onboard` offers a choice to update specific sections per
  the skill's existing "update specific sections" path.
- **Override body drift.** If the plugin agent body changes after an
  override is generated, the override can become stale. **Mitigation:**
  The header hash detects drift on the next `/onboard` run and asks the
  user to regenerate. The README section documents the regeneration
  option.
- **Two sources of truth.** Plugin files and override files both declare
  an agent with the same name. **Mitigation:** Override files include a
  header comment naming them as generated overrides and pointing at the
  plugin file as the source. Users who open the override by accident
  see the breadcrumb immediately.
- **Interview fatigue.** Adding a step extends the onboarding interview
  by up to three questions. **Mitigation:** The step is gated by a
  single yes/no preamble; users who answer no proceed immediately to
  Summary. The three per-agent questions only fire if the user opted in.

## Migration Plan

No migration required; the change is additive.

- Existing `agents/*.md` plugin files are unchanged.
- Existing `.claude/agent-memory/engineering-leaders/PROJECT.md` is
  unchanged.
- Projects that already have a hand-written `.claude/agents/<agent>.md`
  are detected on the first `/onboard` run after this change ships; the
  skill shows the existing file and asks whether to leave it alone or
  replace it with a generated override.
- Rollback is removal of the onboarding step, removal of the README
  section, and deletion of any override files the skill generated. No
  plugin-level data is lost.

## Open Questions

- Should the prompt offer a "same as my main loop" convenience option
  that reads the user's current Claude Code model and biases all three
  agents accordingly? Deferred: the mechanism to read the current main
  model from a skill is not yet established and the explicit per-agent
  trade-off is more educational on first run.
- Should the set of prompted agents expand beyond Tech Lead, Chief
  Architect, and Product Owner? Deferred: revisit once usage data shows
  which other agents are most often overridden. The override mechanism
  itself supports any agent; only the prompt list is scoped.
- Should `.claude/agents/` overrides be generated lazily, only the first
  time an agent is invoked after `/onboard`? Deferred: eager generation
  during onboarding is simpler and fits the existing `/onboard` model
  of writing config at interview time.
