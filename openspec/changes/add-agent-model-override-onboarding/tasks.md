# Tasks

## 1. Verify override mechanism

- [ ] 1.1 Confirm that Claude Code loads an agent definition from
      `.claude/agents/<name>.md` in the project when a plugin of the same
      name is installed, and that the project file's `model:` value wins
- [ ] 1.2 Document the verification result in a comment at the top of the
      onboarding skill's Model Selection step so the fallback path is
      discoverable
- [ ] 1.3 If verification fails, design the manual-fallback copy that the
      onboarding step will emit instead of writing override files

## 2. Update `/onboard` skill

- [ ] 2.1 Insert a new step "Model Selection" between the existing
      Specialist Discovery step and the Summary step
- [ ] 2.2 Add a preamble yes/no question that gates the three per-agent
      prompts so the step is skippable with a single answer
- [ ] 2.3 Add a bounded multiple-choice prompt for Tech Lead using the
      trade-off framing (faster and lower cost / default / higher-quality
      planning), not a model name
- [ ] 2.4 Add the same style of prompt for Chief Architect and Product Owner
- [ ] 2.5 On non-default answers, copy the plugin agent's current body into
      `.claude/agents/<agent-name>.md` with only the `model:` field changed
- [ ] 2.6 Write a header comment in each generated override that records
      (a) that the file was generated by `/onboard`, (b) the plugin agent
      name and path it was copied from, and (c) a hash of the plugin agent
      body at generation time
- [ ] 2.7 Detect a pre-existing `.claude/agents/<agent-name>.md` before
      writing and ask the user whether to leave it unchanged or replace
      it with a generated override
- [ ] 2.8 Update the skill's Summary to list which agents received
      overrides and which did not so the state is legible
- [ ] 2.9 If verification from task 1 failed, fall back to emitting the
      exact frontmatter edit to apply to the plugin agent file along with
      a warning that the edit will be lost on plugin update

## 5. Regeneration detection

- [ ] 5.1 On a subsequent `/onboard` run, detect whether a generated
      override's recorded body hash still matches the current plugin
      agent body
- [ ] 5.2 If the hashes differ, show the user the plugin agent diff and
      offer a choice: regenerate the override (losing local edits) or
      keep the current override (accepting body drift)
- [ ] 5.3 Record the user's choice in the Summary so later runs can
      distinguish "never prompted" from "prompted and declined"
